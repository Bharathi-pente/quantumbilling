## Description

**As an ORG_ADMIN or SUPER_ADMIN**, I want to view, download, and manage the presentation and payment of invoices generated by the billing engine, so that I can track what each customer owes, monitor payment status, follow the dunning process, and see all corrections (credit and debit notes) applied to issued invoices.

**Architecture boundary (ADR-001 ┬¦2.4, ┬¦3):** Invoice **generation belongs exclusively to the Go billing worker**. The worker composes each invoice from two authoritative inputs ŌĆö usage aggregated from ClickHouse and plans/contracts/rate cards from the Postgres control plane ŌĆö and writes the results to the `billing` schema. This story is the **NestJS read/present/pay surface** over those tables. NestJS never computes line items, never aggregates usage, and never writes financial artifacts (one-writer rule: the Go billing worker is the sole writer of `billing.invoices`, `billing.invoice_line_items`, `billing.credit_notes`, `billing.payments`, and `billing.invoice_status_history`; NestJS is a reader/presenter). Mutating actions exposed in the UI (pay, void, record wire payment) are **commands forwarded to the Go billing worker**, which performs the write.

Key concepts:
- An **Invoice** is generated by the Go billing worker per **Subscription** for a **Customer** within an **Organization**, on the subscription's anniversary window (ADR-001 ┬¦3.1)
- **One subscription = one recurring invoice per billing period**, unless the customer belongs to a billing group (CR-8), in which case one consolidated invoice rolls up multiple subscriptions with per-line attribution
- Line items carry an explicit `line_type`: `BASE_FEE`, `USAGE`, `OVERAGE`, `COMMIT_TRUE_UP`, `SEAT`, `ADJUSTMENT`
- Invoice statuses: `draft` ŌåÆ `pending` ŌåÆ `paid` / `overdue` ŌåÆ `voided`
- **Issued invoices are IMMUTABLE.** All corrections happen via **credit notes** (CR-4) ŌĆö never by editing or regenerating an issued invoice
- Payment auto-collection via Stripe is the default (CR-6); manual payment recording remains for wires/checks
- Customers can view and pay their invoices in the customer portal

**State Definitions:**
| State | Description |
|-------|-------------|
| `draft` | Generated by the worker at anniversary; held for a 24ŌĆō48h grace window for late-arriving events (ADR-001 ┬¦3.1) |
| `pending` | Finalized by the worker and sent to the customer; auto-collection attempted (CR-6); awaiting payment |
| `paid` | Payment received in full, invoice closed |
| `overdue` | Payment past due date, dunning process activated |
| `voided` | Invoice canceled (draft-stage error or write-off); no payment will be collected |

All state transitions are written by the Go billing worker to `billing.invoice_status_history`; the UI renders this as a status timeline. **Once an invoice leaves `draft`, its line items and totals never change** ŌĆö post-finalization usage events become `ADJUSTMENT` lines on the next invoice or credit notes (CR-1), never edits.

## Acceptance Criteria

### Invoice Data (read model)

1. Every invoice presented by this story was written by the Go billing worker; NestJS performs no invoice math and no aggregation.
2. Invoice is tied to: `subscription_id`, `customer_id`, `org_id`, `period_start`, `period_end` (the subscription's anniversary window, not the calendar month).
3. Invoice number format: `INV-{YYYY}-{MM}-{NNN}` (e.g., `INV-2026-01-001`), assigned by the worker.
4. Invoice detail exposes: `invoice_number`, `status`, `period_start`/`period_end`, `subtotal`, `credits_applied`, `tax_rate`, `tax_amount`, `total`, `currency`, `due_date`, `issued_at`, `paid_at`.
5. Each line item exposes `line_type` (`BASE_FEE|USAGE|OVERAGE|COMMIT_TRUE_UP|SEAT|ADJUSTMENT`), description, quantity, unit price, amount, meter/model attribution, and its `rate_source`/`rate_source_id` (the ADR-001 ┬¦3.3 rating-waterfall provenance).
6. Issued invoices are immutable. No endpoint in this story updates an invoice's line items, amounts, or periods. Attempting to do so returns `INVOICE_IMMUTABLE` (409).

### Invoice List View (ORG_ADMIN / SUPER_ADMIN)

7. Invoice list shows all invoices for the organization's customers (or all orgs for SUPER_ADMIN).
8. List columns: Invoice #, Customer, Subscription/Plan (or "Consolidated" when `group_id` is set), Issued Date, Due Date, Total, Status, Actions.
9. Status filter tabs: All, Draft, Pending, Paid, Overdue, Voided.
10. Organization filter (SUPER_ADMIN only) and customer filter (ORG_ADMIN).
11. Date range filter for invoice search.
12. Each row is clickable ŌåÆ opens invoice detail.

### Invoice Detail View

13. Invoice header: invoice number, status badge, issued date, due date, period window, customer and organization names.
14. Line items table **grouped by `line_type`**, in order: Base Fee, Seats, Usage, Overage, Commit True-Up, Adjustments ŌĆö each group subtotaled.
15. `ADJUSTMENT` lines carry an explanatory label: "Prior-period adjustment ŌĆö late or corrected usage from a previous, already-issued invoice (issued invoices are never modified; corrections appear here or as credit notes)."
16. Summary section: Subtotal, Credits Applied (negative), Tax (rate %), Tax Amount, **Total**.
17. **Credit Notes section**: all credit and debit notes linked to this invoice ŌĆö note number, kind (`credit`/`debit`), amount, reason, status (`draft|issued|applied|refunded`), and link to the originating re-rating run where present (CR-1/CR-4).
18. **Status timeline** rendered from `billing.invoice_status_history`: each transition with old status, new status, actor, and timestamp.
19. Payment information: collection mode (`auto_charge|manual|wire`), payment method used (if paid), payment date, and partial-payment history.
20. **Consolidated invoices (CR-8):** when `group_id` is set, the detail view shows the billing group name and each line item displays its constituent subscription attribution (`subscription_id` ŌåÆ plan name), with an optional group-by-subscription toggle.
21. Actions: Download PDF, Send Reminder (if overdue), Request Void (if draft/pending/overdue ŌĆö executed by the worker).

### Invoice PDF

22. Invoice can be downloaded as a PDF.
23. PDF includes: company header, invoice number, customer and org details, line items grouped by type, totals, credit notes issued against the invoice, payment instructions.
24. PDF follows a professional invoice template; the rendered PDF reflects exactly the worker-written data (no recomputation).

### Payment ŌĆö Auto-Collection & Pay Now (CR-6)

25. Auto-collection is the default: on finalization, the worker charges the customer's default Stripe payment method; this story displays the resulting payment records and failure reasons.
26. "Pay Now" in the portal creates a Stripe payment for the invoice balance (card or ACH/SEPA) via NestJS Ōåö Stripe; the confirmed payment is recorded by the billing worker and the invoice moves to `paid`.
27. Manual payment recording remains for wires/checks: ORG_ADMIN submits amount, method, date; the command is forwarded to the worker, which writes the `billing.payments` row (`collection_mode = wire|manual`).
28. Partial payments are supported (invoice remains `pending` until fully paid; balance shown = total ŌłÆ sum of completed payments).
29. Overpayment creates a credit on the customer (worker writes the credit ledger entry).
30. Failed auto-collection feeds the dunning state machine and smart-retry schedule (worker-owned).

### Credit Notes (CR-4)

31. Credit and debit notes are listed per customer and per invoice; each shows note number, kind, amount, currency, reason, status (`draft|issued|applied|refunded`), and created date.
32. Credit notes are the **only** correction mechanism for issued invoices ŌĆö the UI never offers "edit invoice" or "regenerate invoice".
33. A credit note linked to a re-rating run (CR-1) displays the run's trigger (late events / rate change / correction) and period.
34. Credit note PDF download is available alongside the invoice PDF.

### Void Invoice

35. Only `draft`, `pending`, or `overdue` invoices can be voided; the request is forwarded to the billing worker, which performs the transition and writes the status history row.
36. Voiding cancels collection ŌĆö no payment will be collected.
37. Voided invoices cannot be un-voided. Corrections to a paid invoice go through credit notes, not voiding.
38. Voiding triggers webhook `invoice.voided`.

### Overdue & Dunning

39. When an invoice's due date passes without full payment, the worker transitions it to `overdue` (visible immediately in this UI).
40. Dunning reminders are sent per the dunning schedule (worker-owned; see QB-STORY-012).
41. ORG_ADMIN can manually send a payment reminder from the detail view.
42. After max dunning attempts, the invoice is written off (voided) or escalated; the status timeline records it.

### Customer Portal View (CUSTOMER)

43. CUSTOMER can view all invoices for their customer account in the portal.
44. Portal list shows: invoice number, total, status, due date, PDF download.
45. CUSTOMER can pay pending invoices via "Pay Now" (Stripe card or ACH).
46. Portal shows credit notes issued to the customer.
47. CUSTOMER cannot void or edit invoices.

## Invoice Line Item Types

Line items are composed by the Go billing worker (ADR-001 ┬¦3); this story only displays them, grouped by type.

| `line_type` | Computed by worker from | Example |
|-------------|------------------------|---------|
| **BASE_FEE** | Subscription + plan price, prorated | Enterprise Plan: $499/month |
| **USAGE** | ClickHouse period aggregation ├Ś resolved rate (┬¦3.3 waterfall) | GPT-4 Input Tokens: 156M @ contract rate |
| **OVERAGE** | `max(0, usage ŌłÆ included units)` ├Ś overage rate | Overage Usage: $9,443.80 |
| **COMMIT_TRUE_UP** | `max(0, commit_amount ŌłÆ eligible spend over the contract term)`; eligible spend is USAGE + OVERAGE only, and this line appears only on the final invoice of the contract term | Commit shortfall true-up: $2,000 |
| **SEAT** | Seat count ├Ś seat price, with seat-change proration (CR-3) | 25 seats ├Ś $20 |
| **ADJUSTMENT** | Prior-period corrections: late/corrected events, proration deltas | Prior-period adjustment (Feb): $84.20 |

Every line records its `rate_source`/`rate_source_id` from the rating waterfall (contract rate ŌåÆ pinned rate-card version ŌåÆ plan pricing model), supporting CR-1 reproducibility. Unrated usage never appears silently ŌĆö it is flagged on the worker's rating-exceptions report. Promotional/prepaid credits appear as invoice-level `credits_applied`, not line items; post-issue corrections appear as **credit notes**.

## State Machine ŌĆö Invoice Lifecycle

All transitions are executed by the Go billing worker and recorded in `billing.invoice_status_history`. NestJS displays the timeline and forwards void requests.

### State Transitions

| From State | To State | Trigger (worker-executed) |
|-----------|----------|---------------------------|
| (none) | `draft` | Worker generates invoice at anniversary + grace window |
| `draft` | `pending` | Worker finalizes and sends; auto-collection attempted (CR-6) |
| `draft` | `voided` | Void of a draft-stage error (void/reissue flow, CR-4) |
| `pending` | `paid` | Payment received (full) ŌĆö auto-charge, Pay Now, or wire |
| `pending` | `overdue` | Due date passed without full payment |
| `pending` | `voided` | Void requested (via UI command) |
| `overdue` | `paid` | Payment received (full) |
| `overdue` | `voided` | Write-off after dunning fails |

Terminal correctness rule: `paid` and `voided` invoices are never reopened or edited ŌĆö discrepancies are settled with credit or debit notes (CR-4).

### Scheduled Processing (Go billing worker ŌĆö out of scope for this story)

Invoice generation, the draft grace window and finalization, the overdue check, auto-collection retries, and dunning escalation all run inside the Go billing worker per ADR-001 ┬¦3.1 and CR-6. **This story defines no cron jobs.** The former NestJS `invoice-generator` cron is deleted.

## Dependencies & Notes for Agent

- **Generation boundary (ADR-001 ┬¦2.4, ┬¦3):** invoice generation, line-item computation, usage aggregation, rating, proration, credit application, and tax calculation all happen in the **Go billing worker**. NestJS must contain **no** invoice math ŌĆö no `SUM(value) ├Ś rate`, no period aggregation, no credit FEFO logic. If a number is displayed, it was read from a `billing.*` column.
- **One-writer rule (ADR-001 ┬¦2):** the Go billing worker is the sole writer of `billing.invoices`, `billing.invoice_line_items`, `billing.credit_notes`, `billing.payments`, and `billing.invoice_status_history`. NestJS Prisma access to these tables is SELECT-only; enforce with a read-only role or Prisma middleware guard.
- **Mutations as commands:** Void and wire/manual payment recording are forwarded to the worker's command API (`BILLING_WORKER_BASE_URL`, service-to-service auth). The worker validates the transition, writes the rows, and emits the webhook. NestJS surfaces `BILLING_WORKER_UNAVAILABLE` on delivery failure.
- **Pay Now (CR-6):** NestJS creates the Stripe PaymentIntent/Checkout for the invoice's open balance; the Stripe confirmation webhook is consumed by the worker, which records the payment and transitions the invoice. Auto-collection on finalization is the default; manual recording stays for wires/checks only.
- **Immutability:** issued invoices are never edited or regenerated. Corrections = credit notes (CR-4), driven by re-rating runs (CR-1). Post-finalization usage becomes `ADJUSTMENT` lines on the next invoice. Surface both clearly in the UI (adjustment explainer, credit-notes section).
- **Billing periods:** `period_start`/`period_end` are the subscription's anniversary window (ADR-001 ┬¦3.1), not calendar months. Display the window verbatim; never recompute it.
- **Consolidated invoices (CR-8):** when `group_id` is non-null, join `billing.billing_groups` for the group name and use `invoice_line_items.subscription_id` for per-line attribution; support the group-by-subscription view.
- **Snapshot refs (CR-1):** `rate_card_version_id`, `plan_version_id`, `aggregation_watermark` on the invoice are reproducibility metadata; show them in an admin-only "audit" expander, never edit them.
- **Vocabulary (ADR-001 ┬¦2.1):** the payer is the **customer** (`customer.customers`); **end_user** is the consumption sub-identity. Do not use "tenant"/"user" for these concepts anywhere in code or UI copy.
- **Invoice PDF:** rendered by NestJS from worker-written data (PDFKit/Puppeteer or a dedicated service); stored at `INVOICE_STORAGE_PATH`, served via `GET /invoices/:id/pdf`. Same pipeline for credit-note PDFs.
- **Dunning integration:** overdue invoices feed the worker's dunning state machine (see Dunning user story QB-STORY-012); this UI displays reminders sent and allows a manual re-send.
- **Audit logging:** invoice state changes are already audited in `billing.invoice_status_history` (worker-written); log UI-initiated actions (void request, reminder, Pay Now) to `platform.audit_logs` with actor and timestamp.
- **Access Control:** ORG_ADMIN sees only their org's customers' invoices; CUSTOMER sees only their own; SUPER_ADMIN sees all. Scope is derived from the Keycloak JWT in the NestJS BFF.
